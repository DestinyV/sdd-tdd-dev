# 前端基础设施任务指南

## ⭐🆕 全栈模式：前端开发依赖接口契约

当项目涉及前后端都改动时，**前端任务必须在接口契约确认后才能开始**：

```
前置条件（由后端完成）：
  ✅ 数据库设计已确认
  ✅ 接口契约已定义（API端点、请求/响应、错误码）
  ✅ Mock 数据已生成
  ✅ 前后端双方已确认接口契约
    ↓

前端核心开发（使用 Mock 数据，不依赖后端实际实现）：
  T-FE1: 组件层实现（UI/Props/状态）
  T-FE2: API 集成层（使用 Mock 数据）
  T-FE3: 状态管理
    ↓

前端测试编写：
  T-E2E: E2E 测试（使用 Mock 数据）
  T-VISUAL: 视觉回归测试
  T-COMPONENT: 组件 UI 测试
    ↓

联调阶段（后端 API 实现后）：
  T-INT: 替换 Mock 为真实 API，联调验证
```

> **⚠️ 关键约束**：前端不得在没有接口契约的情况下自行定义 API 调用方式。
> 所有 API 调用必须基于 design.md 中接口契约章节定义的请求参数和响应体。
> 前端开发使用 Mock 数据独立进行，不阻塞开发进度。

## 前端基础设施任务清单

| 任务类别 | Task 示例 | 依赖关系 | 优先级 |
|---------|----------|---------|--------|
| 🆕 接口契约确认 | 确认API定义、Mock数据可用 | design.md 接口契约章节完成后 | **最高（先执行）** |
| 构建优化 | 代码分割配置、压缩优化、Tree Shaking | 代码开发并行 | 中 |
| CDN配置 | 静态资源上传CDN、缓存策略设置 | 构建配置完成后 | 低 |
| 错误追踪 | Sentry/监控SDK接入 | 与业务逻辑并行 | 中 |
| 性能监控 | Core Web Vitals采集、首屏计时 | 与业务逻辑并行 | 中 |
| 用户埋点 | PV/UV、按钮点击、业务路径埋点 | UI组件开发后 | 中 |
| 兼容性 | Polyfill配置、浏览器测试 | 代码开发后 | 中 |
| **浏览器 E2E 测试** ⭐ | Playwright 端到端测试编写（使用 Mock） | 组件实现后并行 | 高 |
| **视觉回归测试** ⭐ | 关键页面截图对比配置 | 组件实现后并行 | 高 |
| **组件 UI 测试** ⭐ | 组件级交互测试编写 | 组件实现后并行 | 高 |
| **联调验证** ⭐ | 替换 Mock 为真实 API | 后端 API 实现后 | 高 |

## 前端测试任务说明

每个前端 UI 组件 Task 都应关联对应的浏览器测试 Task：

```
T1: 登录组件实现 → 关联 BT-1: 登录 E2E 测试
T2: 表单验证逻辑 → 关联 BT-2: 表单错误展示 E2E
T3: Dashboard 页面 → 关联 BT-3: Dashboard 视觉回归
```

测试 Task 可以和对应组件 Task 在同一批次中由同一子代理完成。

## 前端上线执行顺序

```
上线前准备（独立于开发Task）：
  T-BUILD: 生产构建验证
  T-CDN: CDN域名和缓存配置确认
  T-COMPAT: 浏览器兼容性验证

上线执行（严格按顺序，不可并行）：
  T-UPLOAD: 构建产物上传CDN
  T-DEPLOY: 部署HTML入口文件
  T-CLEAR: 清除CDN缓存（如需要）
  T-VERIFY: 线上功能验证
  T-MONITOR: 监控数据确认（JS错误率、加载时间）
```

## 前端回滚任务

```
T-FE-ROLLBACK:
  - 上一版本构建产物确认
  - HTML入口文件回滚
  - CDN缓存清除
  - 回滚验证
```

## 前端任务依赖图

### 全栈模式（前后端都改动）⭐🆕

```
前置条件（接口契约确认后）：
  ✅ 接口契约已确认（API端点、请求/响应、错误码）
  ✅ Mock 数据已生成
    ↓

Batch 0（构建配置）：T-BUILD-CONF(构建配置) → 执行
  ↓

Batch 1（核心开发，使用 Mock 数据）：
  T-FE1(组件层), T-FE2(API集成+Mock), T-FE3(状态管理) → 并行
    ↓

Batch 2（增强）：
  T-TEST(E2E测试+Mock), T-VISUAL(视觉回归), T-MON(错误追踪+性能) → 并行
    ↓

Batch 3（联调，后端 API 就绪后）：
  T-INT(替换 Mock 为真实 API，联调验证) → 执行
    ↓

Batch 4（部署）：
  T-CDN(上传CDN) → 执行
    ↓

上线阶段（严格按顺序）：
  T-UPLOAD → T-DEPLOY → T-CLEAR → T-VERIFY → T-MONITOR
```

### 仅前端模式

```
Batch 0（准备阶段）：T-BUILD-CONF(构建配置) → 执行
  ↓
Batch 1（核心开发）：T1(组件层), T2(状态管理), T3(API集成) → 并行
  ↓
Batch 2（增强）：T4(样式优化), T5(错误追踪), T6(性能监控) → 并行
  ↓
Batch 3（部署）：T-CDN(上传CDN) → 执行
  ↓
上线阶段（严格按顺序）：T-UPLOAD → T-DEPLOY → T-CLEAR → T-VERIFY → T-MONITOR
```

## 前端任务类型

- **🆕 接口契约确认** ⭐：确认 API 定义、Mock 数据可用（全栈模式下最先完成）
- **🆕 联调验证** ⭐：替换 Mock 为真实 API，联调验证
- **🆕 跨项目接口集成** ⭐🆕（多项目模式）：集成上游项目的 API（使用 Mock 先行）
- **🆕 跨项目联调验证** ⭐🆕（多项目模式）：与上游项目真实 API 联调
- **构建配置** ⭐：Webpack/Vite配置优化、代码分割、压缩
- **CDN部署** ⭐：静态资源上传CDN、缓存策略配置
- **前端监控** ⭐：错误追踪SDK接入、性能指标采集、用户行为埋点
- **浏览器 E2E 测试** ⭐：Playwright 端到端测试，覆盖完整用户流程
- **视觉回归测试** ⭐：关键页面/组件截图对比
- **组件 UI 测试** ⭐：组件级别渲染和交互测试

## 多项目协作场景前端任务指南 ⭐🆕

> 当协作模式非 single 时适用。

### 跨项目 API 集成

前端在多项目场景下需要消费多个上游项目的 API：

```
Phase 1（接口契约确认）：
  T-API-CONFIRM: 确认所有上游项目的接口契约
  T-MOCK: 生成各项目的 Mock 数据
    ↓

Phase 2（Mock 驱动开发）：
  T-FE-COMP: 组件层实现（使用 Mock）
  T-FE-API: 集成层实现（对接多项目 API + Mock）
    ↓

Phase 3（联调 - 按项目依赖顺序）：
  上游项目 A 接口就绪 → T-INT-A: 联调项目 A 接口
  上游项目 B 接口就绪 → T-INT-B: 联调项目 B 接口
    ↓

Phase 4（全链路验证）：
  T-E2E: 端到端测试（覆盖跨项目完整链路）
```

### 前端联调验证

| 联调对象 | 上游项目 | 前端任务 | 验证方式 |
|---------|---------|---------|---------|
| 认证接口 | project-a (auth) | T-INT-AUTH | 登录/Token 刷新/鉴权 |
| 业务接口 | project-b (api) | T-INT-BIZ | CRUD 操作验证 |
| 第三方接口 | project-c (external) | T-INT-EXT | 数据集成验证 |
