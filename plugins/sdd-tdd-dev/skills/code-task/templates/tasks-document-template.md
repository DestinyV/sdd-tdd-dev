## 数据库和接口契约确认 ⭐🆕（fullstack 模式时必填）

> **此部分用于确保前后端开发基于同一套数据模型和接口规范。**
> 读取 spec/requirement.md 标签：
> - 开发模式：[fullstack | frontend-only | backend-only]
> - 需要数据库设计：[是 | 否]
> - 需要接口契约：[是 | 否]

### 数据库设计确认 ⭐（当 需要数据库设计=是 时必填）
- [ ] 数据库表结构已在 design.md 中定义
- [ ] DDL 脚本已生成并验证
- [ ] 数据迁移方案已确认
- [ ] 用户已确认数据库设计
- [ ] N/A（不需要数据库设计）

### 接口契约确认 ⭐（当 开发模式=fullstack && 需要接口契约=是 时必填）
- [ ] API 端点列表已在 design.md 中完整定义
- [ ] 请求参数规范已明确（类型、必填、校验）
- [ ] 响应体结构已定义（与数据库字段一致）
- [ ] 错误码定义完整
- [ ] Mock 数据示例已生成
- [ ] 后端已确认可按契约实现 API
- [ ] 前端已确认可按契约实现 UI

### 依赖关系强制校验

| 校验项 | 状态 | 说明 |
|--------|------|------|
| 数据库设计 → 接口契约 | ✅/❌ | 接口字段是否与数据库设计一致 |
| 接口契约 → 前端任务 | ✅/❌ | 前端任务是否依赖已定义的接口 |
| 接口契约 → 后端任务 | ✅/❌ | 后端任务是否实现已定义的接口 |
| 前端任务 ↔ 后端任务 | ✅/❌ | 双方是否基于同一份接口契约 |

> **如果以上任何校验项为 ❌，此任务列表不得进入执行阶段。**

## 并行执行计划

### 全栈模式（前后端都改动）⭐🆕

```
Batch 0（准备阶段，必须串行）：
  T-DB: 数据库迁移脚本编写和验证 → 用户确认
    ↓ (数据库确认完成后)
  T-API-CONTRACT: 接口契约定义 + Mock 数据生成 → 前后端双方确认
    ↓ (接口契约确认后)

Batch 1（核心开发，前后端可并行）：
  前端：T-FE1(组件层), T-FE2(状态管理), T-FE3(API集成) → 使用 Mock 并行
  后端：T-BE1(数据层), T-BE2(服务层), T-BE3(API实现) → 基于数据库实现
    ↓ (等待全部完成)

Batch 2（联调与增强）：
  T-INT: 前后端联调（替换 Mock 为真实 API）
  T-TEST: E2E 测试、集成测试
    ↓ (等待全部完成)

Batch 3（部署准备）：
  T-DEPLOY: 构建配置、CDN、监控
```

### 仅前端/仅后端模式

- Batch 1（0h）：T1, T2, T3 → 3 parallelism
- Batch 2（4h）：T4, T5 → 2 parallelism
- Batch 3（8h）：T6 → 1 parallelism
