# 架构审查清单

> code-designer 阶段设计审查和 code-execute 阶段代码审查使用

## 1. 层依赖检查

| # | 检查项 | 严重级别 | 说明 |
|---|--------|---------|------|
| 1.1 | 上层是否只依赖下层？ | 🔴 严重 | 表现层→业务层→数据层，禁止反向依赖 |
| 1.2 | 是否有跨层跳跃？ | 🟡 重要 | 如表现层直接调用数据层，应经过业务层 |
| 1.3 | 是否有循环依赖？ | 🔴 严重 | A→B→A 必须打破，可通过接口或事件解耦 |
| 1.4 | 领域层是否依赖外部框架？ | 🔴 严重 | 领域层应独立于数据库、Web框架等 |
| 1.5 | 依赖是否通过接口而非具体类？ | 🟡 重要 | 高层模块应依赖抽象，非具体实现 |

## 2. 设计模式检查

| # | 检查项 | 严重级别 | 说明 |
|---|--------|---------|------|
| 2.1 | 模式选择是否恰当？ | 🟡 重要 | 是否适合当前场景，是否有更简单的方案 |
| 2.2 | 是否过度设计？ | 🟡 重要 | 简单 CRUD 使用多种模式 = 过度设计 |
| 2.3 | 是否存在模式贫血？ | 🟡 重要 | 实体类只有 getter/setter 而无业务行为 |
| 2.4 | Singleton 是否滥用？ | 🟡 重要 | 应优先使用 DI，仅在真正全局唯一时使用 |

## 3. SRP（单一职责）检查

| # | 检查项 | 严重级别 | 说明 |
|---|--------|---------|------|
| 3.1 | 类/模块是否职责单一？ | 🟡 重要 | 一个类有多个变化理由应拆分 |
| 3.2 | 函数是否超过 30 行？ | 🟢 建议 | 长函数应考虑拆分为多个小函数 |
| 3.3 | 函数名是否暗示多职责？ | 🟡 重要 | 如 `validateAndProcessAndSave` 应拆分 |
| 3.4 | 类文件是否超过 300 行？ | 🟢 建议 | 大文件应考虑按职责拆分 |

## 4. DIP（依赖倒置）检查

| # | 检查项 | 严重级别 | 说明 |
|---|--------|---------|------|
| 4.1 | 是否依赖接口而非具体类？ | 🔴 严重 | `new MySQLRepo()` → 通过构造函数注入接口 |
| 4.2 | 是否使用构造函数注入？ | 🟡 重要 | 避免属性注入和方法注入（除非有明确理由） |
| 4.3 | 外部依赖是否有适配器？ | 🟡 重要 | 第三方 API 调用应通过适配器隔离 |
| 4.4 | 是否避免了 Service Locator？ | 🟡 重要 | Service Locator 隐藏依赖，难测试 |

## 5. 组件质量检查

| # | 检查项 | 严重级别 | 说明 |
|---|--------|---------|------|
| 5.1 | 组件大小是否合理？ | 🟢 建议 | Template >150 行或 Script >200 行应考虑拆分 |
| 5.2 | Props 是否超过 5 个？ | 🟢 建议 | 过多 Props 应考虑组合或拆分 |
| 5.3 | 组件是否可独立测试？ | 🟡 重要 | 不能直接测试的组件说明耦合过重 |
| 5.4 | 是否存在重复 UI 代码？ | 🟡 重要 | 相同/相似 UI 应提取为共享组件 |
| 5.5 | 是否使用组合而非继承？ | 🟡 重要 | 继承链超过 3 层应考虑改为组合 |

## 6. 领域模型检查

| # | 检查项 | 严重级别 | 说明 |
|---|--------|---------|------|
| 6.1 | 聚合边界是否清晰？ | 🟡 重要 | 聚合内对象有强一致性，外部只能通过聚合根 |
| 6.2 | 值对象是否不可变？ | 🔴 严重 | 值对象一旦创建属性不应变化 |
| 6.3 | 仓储是否定义了接口？ | 🟡 重要 | Repository 应定义在领域层，实现在基础设施层 |
| 6.4 | 实体是否有业务行为？ | 🟡 重要 | 实体不应只是数据容器（避免贫血模型） |
| 6.5 | 领域语言是否统一？ | 🟢 建议 | 类名/方法名应使用业务术语，非技术术语 |

## 7. 反模式检测

| # | 检查项 | 严重级别 | 说明 |
|---|--------|---------|------|
| 7.1 | 上帝对象（God Class/Object）？ | 🔴 严重 | 一个类/模块承担了过多职责 |
| 7.2 | 意大利面条代码？ | 🔴 严重 | 无结构、无抽象、深度嵌套的逻辑 |
| 7.3 | 紧耦合模块？ | 🟡 重要 | 模块间直接依赖具体实现，无法独立替换 |
| 7.4 | 金锤子（用模式解决一切）？ | 🟡 重要 | 简单问题使用复杂设计模式 |
| 7.5 | 过早抽象？ | 🟢 建议 | 第一次出现就提取抽象，应等第二次出现 |
| 7.6 | 上帝函数（长函数）？ | 🟡 重要 | 函数超过 30 行，包含多个逻辑块 |

## 8. 注释和实现检查

| # | 检查项 | 严重级别 | 说明 |
|---|--------|---------|------|
| 8.1 | 是否只写注释不实现？ | 🔴 严重 | 函数体只有注释，没有真正的代码实现 |
| 8.2 | 是否存在空函数体 + 注释？ | 🔴 严重 | 函数体为空或只有 TODO/注释 |
| 8.3 | 是否用注释掉的代码替代实现？ | 🔴 严重 | 注释掉的 fetch/API 调用没有真正执行 |

---

## 使用指南

- **🔴 严重**：必须修复，不得进入下一阶段
- **🟡 重要**：应修复，如有合理理由可暂保留并记录 TODO
- **🟢 建议**：建议优化，可根据实际情况决定是否处理
