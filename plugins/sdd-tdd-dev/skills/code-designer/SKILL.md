- code-designer **不执行**

  - 关键实现决策及原因
  - 样式/交互方案

# code-designer

## 职责

根据规范和用户需求，进行代码设计分析和架构规划。支持全栈开发的各类需求：

1. 读取 `spec-dev/{requirement_desc_abstract}/spec/` 中的规范文档
2. 分析用户需求，识别对应的设计模式和架构模式
3. 查看参考实现（前端组件、后端服务、数据库设计等）
4. 进行需求-参考差异分析
5. 生成详细的设计方案文档到 `spec-dev/{requirement_desc_abstract}/design/design.md`
6. 包括技术栈选型、依赖关系、部署考虑、性能优化等方面

**输出路径**：`spec-dev/{requirement_desc_abstract}/design/design.md`

---

## 工作流程

### 阶段1: 需求确认和项目理解

与用户对话，理解需求，并根据需要分配 **code-architect** 和 **code-explorer** agent 来深度理解项目。

**活动**：
- 理解用户的核心需求（基于 spec-creation 的规范文档）
- 读取 design-spec/README.md，识别对应的设计模式和参考组件
- 一次一个问题，确认用户需求

**关键澄清点**（详见完整列表：[`references/clarification-points.md`](./references/clarification-points.md)）：
- **前端构建和部署** ⭐（前端/全栈场景）
- **前端监控和可观测性** ⭐（前端/全栈场景）
- **前端测试性设计** ⭐（前端/全栈场景）- data-testid 策略、可测试交互路径、视觉回归关键点
- **服务上下游依赖** ⭐
- **数据库表设计改动** ⭐
- **中间件和基础设施** ⭐
- **项目配置** ⭐
- **日志规范** ⭐
- **上线方案** ⭐
- **监控方案** ⭐
- **后端API依赖**
- **数据模型清晰**
- **集成依赖**

- **分配 Agent 了解项目**（根据情况选择）：
  - **主要方案**：分配 **code-architect agent**（代码库模式分析 + 架构设计二合一）
    - code-architect 本身会进行代码库模式分析
    - 包括现有代码结构、技术栈、命名约定、设计模式
    - 然后基于分析进行设计
  - **深度方案**（可选）：先分配 **code-explorer agent** 深度追踪现有实现，再分配 **code-architect**
    - 适用于需要详细理解现有功能实现细节的情况
    - code-explorer 和 code-architect 可以**并行执行**（都是分析和设计阶段）
- 整合 Agent 分析结果，形成项目理解报告
- 识别是新增功能、修改现有功能还是优化

**输出**：
- 清晰的需求理解
- 项目理解报告（来自 code-architect，或 code-explorer + code-architect）
- referenceComponentName（如 ProductTable、OrderForm）
- designPattern（如 "列表管理模式"、"表单配置模式"）
- projectRoot 和 designSpecPath

---

### 阶段2: Agent 分析工作

根据阶段1的方案分配相应的 Agent。

**方案A：标准流程（推荐）**
- **分配 code-architect Agent** 进行项目分析和架构设计
- code-architect 会完成：
  - 代码库模式分析（技术栈、框架版本、文件组织、命名约定等）
  - 寻找现有的类似功能以理解已建立的做法
  - 确定可复用的模式
  - 输出架构蓝图和设计方案

**方案B：深度分析流程**（需要详细理解现有实现）
- **并行分配 code-explorer 和 code-architect Agents**：
  - **code-explorer**：
    - 追踪相关功能的执行路径和代码流
    - 映射抽象层（展示层 → 业务逻辑层 → 数据层）
    - 识别使用的设计模式和架构决策
    - 输出完整的功能实现分析
  - **code-architect**：
    - 进行项目模式分析
    - 基于 code-explorer 的分析进行架构设计
    - 输出架构蓝图
  - 两个 Agent **可以并行执行**（都是分析阶段，不互相阻塞）

**汇总**：
- 整合 Agent 分析结果
- 形成统一的项目理解和架构设计
- 为下一阶段的参考组件分析提供基础

---

### 阶段3: 参考组件分析

基于阶段1-2确认的参数和项目理解，分析参考组件的完整实现。

**任务**：
1. 定位参考组件文件（从design-spec中获取路径）
2. 读取参考组件的源代码，理解：
   - 组件结构和Props设计
   - 内部状态管理方式（结合项目的状态管理方案）
   - 样式和交互实现（结合项目的样式系统）
   - 关键的业务逻辑
3. 分析参考组件的设计决策
   - 为什么这样设计Props？
   - 为什么选择这种状态管理方式？
   - 如何与项目的样式系统集成？
   - 关键的交互流程是什么？

---

### 阶段4: 需求-设计差异分析

比对用户需求和参考组件，识别差异和设计重点。

**默认策略**：
- 新功能**默认完全参考现有设计**
- 只在**有差异的地方**列出来让用户确认
- 参考不完整时，主动查找类似的设计模式

**分析维度**：
1. **功能范围** - 新功能与参考相比，增删改了什么？
2. **数据结构** - 需要什么新的数据字段？
3. **交互流程** - 用户交互流程与参考相比有什么不同？
4. **集成方式** - 如何集成参考组件？需要包装吗？
5. **样式定制** - 是否需要定制样式？

**输出**：
- 差异点清单（如果无差异，则明确说明完全参考）
- 设计决策确认清单

---

### 阶段5: 生成设计方案

<HARD-GATE>
在前端/全栈场景中，未完成测试性设计（data-testid 策略、可测试交互路径）
之前，不输出最终的设计方案文档。
无论任务看起来多么简单，此规则都适用。
</HARD-GATE>

基于分析，生成详细的设计文档（design.md）。

**文档结构**（遵循**单一入口 + 内部导航**策略）：

详见完整设计文档模板：[`templates/design-template.md`](./templates/design-template.md)

模板涵盖以下章节：
1. 需求分析（核心功能、用户场景）
2. 设计方案（架构、组件、数据流、服务依赖、数据库、中间件、日志、样式、交互、前端构建、前端监控）
3. 与参考设计的对比
4. 技术方案（依赖库、API、性能、监控、上线方案、前端上线、前端监控）
5. 实现清单
6. 设计决策记录
7. 潜在风险和注意事项
8. 下一步

**生成原则**：
- 基于参考组件的设计，给出完整的技术方案
- 说明清楚每个设计决策的原因
- 所有涉及的代码结构都要在设计阶段确认
- 为Task阶段做好准备

---

### 阶段6: 审批与交付

向用户展示设计方案，获得确认后进入Task阶段。

**审批步骤**：

1. **展示design.md** - 让用户理解整个设计
2. **用户提问和修改** - 如需调整，修改方案后重新审查
3. **确认无误后交付** - 进入code-task阶段

---

## 关键约束

1. **参考优先** - 完全参考现有设计，只说明新增部分
2. **完整性** - 设计要足够详细，能直接指导编码
3. **可验证** - 设计方案中的每个决策都要有原因
4. **前瞻性** - 考虑未来的扩展可能性，但不过度设计

---

## 危险信号

**永远不要**：
- 跳过对参考组件的分析
- 在设计阶段创新，而不是参考现有方案
- 忽视用户反馈，直接进入Task阶段
- 在设计中引入过多新技术选型
- 做出影响大的设计决策而不说明原因
