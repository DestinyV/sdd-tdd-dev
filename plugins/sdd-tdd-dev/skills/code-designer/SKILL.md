  第2步：代码设计规划（Design阶段）
  
  输入：spec-dev/{requirement_desc_abstract}/spec/
  输出：spec-dev/{requirement_desc_abstract}/design/design.md
  
    功能：根据规范和需求进行架构和技术设计分析，生成代码实现方案和设计决策文档。
    支持全栈开发的各类设计（前端UI、后端服务、数据库、微服务、移动应用等）。
    
    设计范围：
    - 前端：组件架构、状态管理、样式方案、交互设计、测试性设计
    - 后端：API设计、数据处理流程、业务逻辑、服务架构
    - 数据库：表结构设计、索引策略、数据一致性、备份方案
    - 微服务：服务边界、通信协议、数据流、部署方案
    - 移动应用：跨平台方案、离线支持、同步机制、性能优化
    
    ## 核心设计原则：数据库先行 + 接口契约驱动
    
    当需求同时涉及前端和后端改动时，必须遵循以下顺序：
    1. **数据库设计先行**：先完成数据库表结构设计，确认后再进行后续步骤
    2. **接口契约对齐**：基于数据库设计定义前后端接口契约（API Spec），双方确认后再各自开发
    3. **前后端并行开发**：接口契约确认后，前后端基于同一份接口规范独立开发
    4. **Mock数据驱动**：前端开发使用接口契约生成 Mock 数据，不依赖后端实际实现
    
    这个流程确保前后端使用同一套数据模型和接口规范，避免独立开发结束后发现双方是两套方案。
  
## 何时使用

**前置条件**：
- ✅ spec-creation 已完成，规范已生成
- ✅ 标准模式：需要独立设计文档
- ⚠️ **快速模式**：如果用户启用了快速模式，code-designer 跳过。设计决策直接内联到 spec 的 scenarios 中。

### 快速模式下的行为

当 spec 标注为快速模式时：
- code-designer **不执行**
- 设计决策由 spec-creation 在生成场景时一并内联到 scenarios 中
- 每个 scenario 增加"设计决策"字段，说明：
  - 组件/模块结构选择
  - 关键实现决策及原因
  - 样式/交互方案
- 简短（2-5 句话），不生成独立的 design.md

# code-designer

## 职责

根据规范和用户需求，进行代码设计分析和架构规划。支持全栈开发的各类需求：

1. 读取 `spec-dev/{requirement_desc_abstract}/spec/` 中的规范文档
2. 分析用户需求，识别对应的设计模式和架构模式
3. 查看参考实现（前端组件、后端服务、数据库设计等）
4. 进行需求-参考差异分析
5. 生成详细的设计方案文档到 `spec-dev/{requirement_desc_abstract}/design/design.md`
6. 包括技术栈选型、依赖关系、部署考虑、性能优化等方面

**输出路径**：`spec-dev/{requirement_desc_abstract}/design/design.md`

---

## 工作流程

### 阶段0: 读取 spec-creation 标签和设计策略选择

**读取 spec/requirement.md 头部的工作模式标签和全栈判定标签**：

```markdown
## 工作模式
- 模式：[standard | quick]
## 全栈模式判定
- 开发模式：[fullstack | frontend-only | backend-only]
- 是否同时涉及前端和后端改动：[是 | 否]
## 数据库设计需求
- 需要数据库设计：[是 | 否]
## 接口契约需求
- 需要接口契约：[是 | 否]
```

<HARD-GATE>
当标签显示 开发模式=fullstack && 需要接口契约=是 时，必须执行接口契约流程。
当标签显示 需要数据库设计=是 时，必须先完成数据库设计再进入接口契约阶段。
</HARD-GATE>

**策略选择**：
| 开发模式 | 需要数据库 | 设计顺序 |
|---------|-----------|---------|
| fullstack | 是 | 数据库设计 → 接口契约 → 前后端各自设计 |
| fullstack | 否 | 接口契约 → 前后端各自设计 |
| frontend-only | - | 前端组件设计 |
| backend-only | 是 | 数据库设计 → 后端服务 + API |
| backend-only | 否 | 后端服务 + API |

当判定为"fullstack"时，后续流程如下：

```
spec → [阶段1: 需求理解] → [阶段2: 数据库设计(按需)] → [阶段3: 接口契约] → [阶段4+: 前后端设计] → [阶段N: 生成方案]
                              ↑ 用户确认                  ↑ 双方确认              ↑ 基于契约各自独立
```

### 阶段1: 需求确认和项目理解

与用户对话，理解需求，并根据需要分配 **code-architect** 和 **code-explorer** agent 来深度理解项目。

**活动**：
- 理解用户的核心需求（基于 spec-creation 的规范文档）
- 读取 design-spec/README.md，识别对应的设计模式和参考组件
- 一次一个问题，确认用户需求

**关键澄清点**（详见完整列表：[`references/clarification-points.md`](./references/clarification-points.md)）：
- **前端构建和部署** ⭐（前端/全栈场景）
- **前端监控和可观测性** ⭐（前端/全栈场景）
- **前端测试性设计** ⭐（前端/全栈场景）- data-testid 策略、可测试交互路径、视觉回归关键点
- **服务上下游依赖** ⭐
- **数据库表设计改动** ⭐
- **SQL DDL/DML 审查** ⭐🆕 - 完整的可执行 SQL 脚本生成、回滚脚本、执行顺序
- **多项目架构** ⭐🆕 - 跨项目依赖、协作链路、项目清单收集
- **设计模式选择** ⭐🆕 - 根据场景推荐合适的设计模式（详见 `references/design-patterns-guide.md`）
- **架构风格选择** ⭐🆕 - 根据项目复杂度选择架构模式（详见 `references/architecture-patterns-guide.md`）
- **领域建模** ⭐🆕（后端/全栈，复杂业务场景） - 提取实体、值对象、聚合根（详见 `references/domain-modeling-guide.md`）
- **组件拆分分析** ⭐🆕 - 何时拆分、如何拆分（详见 `references/component-extraction-guide.md`）
- **中间件和基础设施** ⭐
- **项目配置** ⭐
- **日志规范** ⭐
- **上线方案** ⭐
- **监控方案** ⭐
- **后端API依赖**
- **数据模型清晰**
- **集成依赖**

- **分配 Agent 了解项目**（根据情况选择）：
  - **前置步骤**（必须执行）：要求 agent 先执行「项目文档发现」
    - 读取项目的 CLAUDE.md / AGENTS.md / README.md / ARCHITECTURE.md 等自述文档
    - 提取项目声明的架构模式、模块边界、分层约定、技术栈
    - 文档声明的约定是架构设计的第一优先级约束
    - 详见参考文档：[`../spec-creation/references/project-convention-discovery.md`](../spec-creation/references/project-convention-discovery.md)

**并行探索策略**（双路并行）：
- 同时发射 **code-explorer** 和 **code-architect**（使用 Agent 工具 + `run_in_background: true`）
- **code-explorer**：追踪相关功能的执行路径、映射抽象层、识别设计模式
- **code-architect**：项目模式分析、架构蓝图设计
- 两者**互不阻塞**，等待全部完成后整合

**整合流程**：
1. 同时发射两路 subagent
2. 以 code-architect 的输出为主体（架构决策、蓝图）
3. 用 code-explorer 的发现补充具体代码引用和路径映射
4. 将两者结果合并为统一的项目理解报告

**容错机制**：
- 任一路失败，不阻塞另一路
- code-explorer 失败：以 code-architect 的代码模式分析为基础，标注缺失具体引用
- code-architect 失败：以 code-explorer 的代码分析为基础，由主 agent 补充架构决策
- 两路都失败：回退到主 agent 自行分析

**小型项目策略**：
- 文件数 < 200 的项目仅分配 code-architect agent
- 无需并行发射

- 整合 Agent 分析结果，形成项目理解报告
- 识别是新增功能、修改现有功能还是优化

**输出**：
- 清晰的需求理解
- 项目理解报告（来自 code-architect，或 code-explorer + code-architect）
- referenceComponentName（如 ProductTable、OrderForm）
- designPattern（如 "列表管理模式"、"表单配置模式"）
- projectRoot 和 designSpecPath

### 阶段2: Agent 分析工作

根据阶段1的并行探索策略，执行 agent 分析。

**执行流程**：
- code-explorer 和 code-architect 按阶段1定义的双路并行策略同时发射
- code-explorer 输出：功能实现分析、代码引用、路径映射
- code-architect 输出：项目模式分析、架构蓝图
- 等待全部完成后整合：以 code-architect 为主体，用 code-explorer 补充具体引用

**多项目并行探索** ⭐🆕：
当需求涉及多个项目/仓库时：
- 对每个项目独立发射分析 subagent（并行执行）
- 每个 subagent 负责收集该项目的技术栈、目录结构、现有模式
- 等待全部完成后，构建项目依赖图
- 主 Agent 负责汇总各项目信息、识别跨项目共享依赖和接口

**容错机制**：
- 任一路失败不阻塞其他路，标注"⚠️ 该路分析失败"
- 全部失败时回退到主 agent 自行分析

### 阶段2.2: 领域建模（复杂业务场景必执行）⭐🆕

**适用场景**：后端/全栈项目，业务逻辑复杂，涉及多领域实体和跨领域协作。

**参考指南**：[`references/domain-modeling-guide.md`](./references/domain-modeling-guide.md)

**执行步骤**：
1. 从 spec 场景中提取所有业务名词
2. 分类为实体、值对象、聚合根
3. 划定聚合边界（强一致性范围）
4. 提取业务行为（从 WHEN-THEN 场景）
5. 定义仓储接口（领域层）和识别应用服务

**输出**：
- 领域模型清单（实体、值对象、聚合根）
- 聚合关系图
- 领域服务清单

### 阶段2.3: 架构原则应用 ⭐🆕

**参考指南**：
- SOLID 原则：[`references/solid-principles-guide.md`](../code-execute/references/solid-principles-guide.md)（位于 code-execute）
- 架构模式：[`references/architecture-patterns-guide.md`](./references/architecture-patterns-guide.md)
- 依赖注入：[`references/dependency-injection-guide.md`](../code-execute/references/dependency-injection-guide.md)（位于 code-execute）

**执行内容**：
1. 根据项目复杂度选择架构模式（Clean/Hexagonal/Layered）
2. 定义各层之间的依赖规则（上层依赖下层，禁止反向依赖）
3. 识别需要依赖注入的点（外部依赖、可替换组件）
4. 在 design.md 中明确架构模式和依赖规则图

### 阶段2.4: 组件拆分分析 ⭐🆕

**参考指南**：[`references/component-extraction-guide.md`](./references/component-extraction-guide.md)

**前端组件拆分**：
1. 根据拆分触发器判定是否需要拆分
2. 确定组件层级（容器组件 vs 展示组件）
3. 设计 Props/Events 接口
4. 识别可复用的共享组件

**后端代码拆分**：
1. 按职责拆分服务类（避免 God Service）
2. 提取公共中间件和工具函数
3. 定义 Repository 接口

**输出**：
- 组件拆分清单（前端）/ 服务拆分清单（后端）
- 组件/服务依赖关系图

### 阶段2.5: 数据库设计（按需执行）⭐ **新增**

**读取 spec/requirement.md 中的「数据库设计需求」标签**：
- `需要数据库设计: 是` → 执行此阶段
- `需要数据库设计: 否` → 跳过此阶段，直接进入阶段3

**HARD-GATE**（当需要数据库设计时）：
未先完成数据库设计并得到用户确认之前，
不得进入后续的接口契约和组件设计阶段。

**当不需要数据库设计时**，直接进入阶段3（接口契约对齐）。

**设计内容**：
1. **数据模型定义**（继承自 spec/data-models.md，深化为物理设计）
2. **DDL 脚本预生成**
3. **数据迁移方案**
4. **事务和一致性设计**

**输出**：
- 数据库设计文档（写入 design.md 的"数据库设计"章节）
- **数据库设计确认清单**（需用户明确确认后方可进入下一阶段）

**用户确认点**：
```
数据库设计已完成，请确认：
[ ] 表结构设计是否完整且符合需求？
[ ] 字段类型和约束是否正确？
[ ] 索引策略是否覆盖主要查询场景？
[ ] 数据迁移方案是否可行？
确认后将继续 → 接口契约设计阶段
```

<HARD-GATE>
当 needs-database=true 时，以下任一项未完成不得进入接口契约和组件设计阶段：
1. 数据库物理设计文档已完成
2. SQL DDL/DML 脚本已生成（含回滚脚本）
3. SQL 方言已确认
4. 用户已确认数据库设计
</HARD-GATE>

### 阶段2.6: 多项目协作协调 ⭐🆕（多项目场景必执行）

**读取 spec/requirement.md 中的「多项目协作」章节**：
- 协作模式：非 single → 执行此阶段
- 协作模式：single → 跳过此阶段

<HARD-GATE>
当 requirement.md 中「涉及项目」非空且非"不涉及"时，
未完成 collaboration-plan.md 生成和用户确认之前，
不得进入后续各项目独立详细设计阶段。
</HARD-GATE>

**设计内容**：
1. **项目依赖图构建**：
   - 解析 requirement.md 中的项目列表和依赖关系
   - 构建有向无环图 (DAG)
   - 验证无循环依赖

2. **跨项目 API 契约定义**：
   - 列出所有跨项目调用的 API 端点
   - 明确提供方和消费方
   - 定义接口版本要求

3. **执行顺序确定**：
   - 对 DAG 进行拓扑排序
   - 确定每个项目的执行阶段
   - 标记依赖门控（gate）

4. **生成协作计划**：
   - 使用 collaboration-plan-template.md 生成 collaboration-plan.md
   - 输出到 `spec-dev/{requirement_desc_abstract}/collaboration-plan.md`

**输出**：
- `spec-dev/{requirement_desc_abstract}/collaboration-plan.md`
- design.md 中增加"多项目协作"章节，引用 collaboration-plan.md

**用户确认点**：
```
多项目协作计划已生成，请确认：
[ ] 项目依赖关系是否正确？
[ ] 执行顺序是否可行？
[ ] 跨项目接口契约是否完整？
[ ] 回滚协调方案是否可执行？
确认后将继续 → 各项目独立详细设计阶段
```

### 阶段3: 接口契约对齐 ⭐ **新增**（前后端都改动时必执行）

<HARD-GATE>
在前后端都改动的场景中，接口契约（API Spec）未得到双方确认之前，
不得进入后续的前端组件设计和后端服务实现设计阶段。
</HARD-GATE>

基于已确认的数据库设计，定义前后端交互的 API 接口契约。

> **🆕 核心原则：参考优先**。接口设计必须参考原有项目的架构、模式，最大化按照原有方式去设计和实现。
> 仅在原有模式无法满足需求时才引入新模式，并需要明确说明原因和差异。

**设计内容**：
1. **原有项目 API 模式分析** ⭐🆕：
   - 现有 API 的组织结构（路由前缀、分层方式）
   - 现有请求/响应的统一包装格式（如 `{ code, data, message }`）
   - 现有的认证和鉴权方式
   - 现有的错误码体系和格式
   - 现有的分页、排序、过滤方式
   - 现有的数据转换和序列化方式

2. **API 端点完整定义**（Method、Path、描述、认证、权限）：
   - **路径命名**：遵循现有项目的路由命名规范
   - **HTTP Method**：遵循现有项目的 RESTful 约定
   - **路由前缀**：与现有项目的 API 版本策略一致

3. **请求参数规范**（含类型、必填、校验、示例）：
   - 参数传递方式（Query/Body/Path）遵循现有模式
   - 参数命名风格（camelCase/snake_case）与现有代码一致
   - 分页/排序参数格式与现有 API 统一

4. **响应体规范**（含数据结构、类型、示例）：
   - **响应包装格式**必须与现有 API 一致
   - 数据字段命名风格与现有代码一致
   - 时间/日期格式与现有 API 统一

5. **错误码定义**（错误码、含义、HTTP状态码、前端处理策略）：
   - **错误码体系**必须与现有项目的错误码格式一致
   - 错误码编号范围不与现有冲突（或明确说明复用）

6. **Mock 数据生成**（供前端开发使用）：
   - Mock 数据格式与定义的接口契约一致
   - Mock 数据值参考现有业务数据的格式和范围

7. **接口版本策略**：
   - 版本化方式与现有项目的 API 版本策略一致
   - 如无现有版本策略，采用项目主流的版本化方式

**前后端对齐确认**：
- ✅ **后端确认**：可以按此契约实现 API，无技术障碍
- ✅ **前端确认**：可以按此契约实现 UI，数据满足展示需求

**输出**：
- 接口契约文档（写入 design.md 的"接口契约"章节）
- **接口契约确认清单**（需前后端双方明确确认后方可进入下一阶段）

**用户确认点**：
```
接口契约已定义完成，请前后端双方确认：
[ ] 后端：可以按此契约实现 API？
[ ] 前端：可以按此契约实现 UI，Mock 数据是否充分？
[ ] 所有字段与数据库设计保持一致？
确认后将进入 → 前后端各自详细设计阶段
```

### 阶段4: 参考组件分析

基于阶段1-2.5确认的参数和项目理解，分析参考组件的完整实现。

**任务**：
1. 定位参考组件文件（从design-spec中获取路径）
2. 读取参考组件的源代码，理解：
   - 组件结构和Props设计
   - 内部状态管理方式（结合项目的状态管理方案）
   - 样式和交互实现（结合项目的样式系统）
   - 关键的业务逻辑
3. 分析参考组件的设计决策
   - 为什么这样设计Props？
   - 为什么选择这种状态管理方式？
   - 如何与项目的样式系统集成？
   - 关键的交互流程是什么？

---

### 阶段5: 需求-设计差异分析

比对用户需求和参考组件，识别差异和设计重点。

**默认策略**：
- 新功能**默认完全参考现有设计**
- 只在**有差异的地方**列出来让用户确认
- 参考不完整时，主动查找类似的设计模式

**分析维度**：
1. **功能范围** - 新功能与参考相比，增删改了什么？
2. **数据结构** - 需要什么新的数据字段？
3. **交互流程** - 用户交互流程与参考相比有什么不同？
4. **集成方式** - 如何集成参考组件？需要包装吗？
5. **样式定制** - 是否需要定制样式？

**输出**：
- 差异点清单（如果无差异，则明确说明完全参考）
- 设计决策确认清单

---

### 阶段6: 生成设计方案

<HARD-GATE>
在前端/全栈场景中，未完成测试性设计（data-testid 策略、可测试交互路径）
之前，不输出最终的设计方案文档。
无论任务看起来多么简单，此规则都适用。
</HARD-GATE>

基于分析，生成详细的设计文档（design.md）。

**文档结构**（遵循**单一入口 + 内部导航**策略）：

详见完整设计文档模板：[`templates/design-template.md`](./templates/design-template.md)

模板涵盖以下章节：
1. 需求分析（核心功能、用户场景）
2. 设计方案（架构模式选择、设计模式应用、分层设计与依赖规则、组件、数据流、服务依赖、数据库、中间件、日志、样式、交互、前端构建、前端监控、组件/模块抽离策略、领域模型设计）
3. 与参考设计的对比
4. 技术方案（依赖库、API、性能、监控、上线方案、前端上线、前端监控）
5. 实现清单
6. 设计决策记录
7. 潜在风险和注意事项
8. 下一步

**生成原则**：
- 基于参考组件的设计，给出完整的技术方案
- 说明清楚每个设计决策的原因
- 所有涉及的代码结构都要在设计阶段确认
- 为Task阶段做好准备

---

### 阶段7: 审批与交付

向用户展示设计方案，获得确认后进入Task阶段。

**审批步骤**：

1. **展示design.md** - 让用户理解整个设计
2. **用户提问和修改** - 如需调整，修改方案后重新审查
3. **确认无误后交付** - 进入code-task阶段

---

## 关键约束

1. **参考优先** - 完全参考现有设计，只说明新增部分
2. **完整性** - 设计要足够详细，能直接指导编码
3. **可验证** - 设计方案中的每个决策都要有原因
4. **前瞻性** - 考虑未来的扩展可能性，但不过度设计
5. **数据库按需执行** ⭐ - 当 spec 标签显示需要数据库设计时才执行，否则跳过
6. **接口契约驱动** ⭐ - fullstack 模式下必须先定义接口契约并得到双方确认，再进行前后端各自设计
7. **禁止独立开发** ⭐ - 不允许前后端在没有共同接口契约的情况下独立进入开发阶段

---

## 危险信号

**永远不要**：
- 跳过对参考组件的分析
- 在设计阶段创新，而不是参考现有方案
- 忽视用户反馈，直接进入Task阶段
- 在设计中引入过多新技术选型
- 做出影响大的设计决策而不说明原因
