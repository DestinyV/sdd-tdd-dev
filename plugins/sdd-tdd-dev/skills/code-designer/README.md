## 📌 快速开始（2分钟）

### 这个 Skill 做什么？

根据规范文档，进行详细的代码设计规划，生成架构方案和设计决策文档。

**核心流程**：
```
分析规范 → 设计方案 → 技术选型 → 关键决策 → 设计文档
```

**前置条件**：
- ✅ spec-creation 已完成，规范文档已生成（spec/目录）

### 基本输出

生成 `spec-dev/{requirement_desc_abstract}/design/design.md`，包含：
- 需求确认
- 架构设计
- 技术方案
- 关键组件设计
- 设计决策记录

## 🔑 关键概念

### 1. 五个设计维度

| 维度 | 内容 | 输出 |
|------|------|------|
| **架构设计** | 整体架构、模块划分、关键组件 | 架构图、组件清单 |
| **技术方案** | 框架选型、库依赖、集成方案 | 技术栈表、依赖清单 |
| **核心算法** | 关键业务逻辑、流程图 | 伪代码、流程图 |
| **数据设计** | 数据模型、API契约、数据流 | 数据模型图、接口定义 |
| **设计决策** | 为什么这样设计、权衡考虑 | 决策记录、权衡分析 |

### 2. 模块划分策略

根据规范中的核心功能，划分模块：

```
用户认证功能
├── 模块1：用户管理
│   ├── 用户注册
│   ├── 用户登录
│   └── 用户信息管理
├── 模块2：权限管理
│   ├── 权限定义
│   └── 权限检查
└── 模块3：会话管理
    ├── Session创建
    └── Token刷新
```

### 3. 技术选型的四个原则

1. **契合性**：技术是否符合项目需求？
2. **生态**：是否有成熟的社区和文档？
3. **性能**：是否能满足性能要求？
4. **学习成本**：团队是否能快速上手？

### 4. 设计决策记录（ADR）

每个重要的设计决策都应该记录：

```markdown
## 决策：使用React而不是Vue

### 背景
需要选择前端框架

### 选项
1. React - 生态丰富、灵活
2. Vue - 学习简单、官方文档好

### 决策
选择React

### 理由
- 团队已有React经验
- 项目复杂度高，需要灵活性
- React生态更丰富（测试库、状态管理等）

### 权衡
- 学习成本相对高（对Vue新手）
- 但长期来看收益更大
```

## ⚠️ 常见陷阱

### 陷阱1: 过度设计

```markdown
# ❌ 不要
- 为不存在的需求设计
- 抽象过度（预留过多接口）
- 过度分层

# ✅ 应该
- 基于现有需求设计
- 保持适度的灵活性
- 清晰的分层，但不过度
```

### 陷阱2: 跳过设计决策记录

```markdown
# ❌ 不要
为什么选择PostgreSQL？
（无说明）

# ✅ 应该
为什么选择PostgreSQL？
- 需要ACID事务（业务规则要求）
- 团队有PostgreSQL经验
- 生态成熟、稳定性高
- 考虑过MySQL，但PostgreSQL功能更全面
```

### 陷阱3: 忽视性能考虑

```markdown
# ❌ 不要
直接返回所有数据，不考虑性能

# ✅ 应该
- API分页（每页20条）
- 数据库查询优化（索引、JOIN优化）
- 缓存策略（Redis缓存热数据）
- 异步处理（后台任务、消息队列）
```

## 📋 工作流检查清单

在进行代码设计时：

```
□ 步骤1: 分析规范
  □ 理解所有核心功能
  □ 识别关键的业务规则
  □ 提取数据模型

□ 步骤2: 架构设计
  □ 划分模块
  □ 设计关键组件
  □ 明确组件间交互

□ 步骤3: 技术方案
  □ 选型前端框架
  □ 选型后端框架
  □ 选型数据库
  □ 选型其他关键库

□ 步骤4: 核心算法
  □ 复杂业务逻辑的伪代码
  □ 关键流程的流程图
  □ 验算边界情况

□ 步骤5: 数据设计
  □ 数据模型设计（表、字段、关系）
  □ API契约定义
  □ 数据流设计

□ 步骤6: 设计决策记录
  □ 记录每个重要的选择
  □ 说明理由和权衡
  □ 文档化

□ 步骤7: 生成设计文档
  □ 生成design.md
  □ 包含所有设计内容
  □ 清晰易懂
```

## 💡 最佳实践

### 1. 从规范出发

设计应该直接来自规范，而不是凭空想象：

```markdown
# 规范中的需求
场景：用户创建订单
- 用户选择商品
- 输入数量和地址
- 系统验证数据
- 创建订单记录
- 发送确认邮件

# 对应的设计
模块：订单管理
组件：OrderForm（表单）、OrderService（业务逻辑）
流程：验证 → 创建 → 通知
```

### 2. 明确的组件边界

每个组件应该有清晰的职责：

```markdown
# OrderForm 组件
职责：
- 收集用户输入
- 显示表单验证错误
- 调用OrderService创建订单

不负责：
- 业务逻辑验证（这是OrderService的职责）
- API调用（这是OrderService的职责）
```

### 3. 可追踪的设计决策

```markdown
## 决策：为什么选择TypeScript？

### 源自规范的需求
- 代码库将持续增长
- 需要长期可维护性
- 团队规模会扩大

### 选项分析
1. JavaScript
   - 优：快速开发
   - 缺：易出现类型错误

2. TypeScript
   - 优：类型安全、代码提示好
   - 缺：学习成本、编译时间

### 决策
选择TypeScript

### 理由
- 类型安全能减少运行时错误
- 代码提示提高开发效率
- 大型项目必需

### 权衡
- 初期学习成本（可接受）
- 构建时间增加（可优化）
```

### 4. 清晰的技术栈表

```markdown
## 技术栈

### 前端
| 层级 | 技术 | 版本 | 选择理由 |
|------|------|------|---------|
| 框架 | React | 18.x | 生态丰富、灵活 |
| 状态管理 | Redux | 4.x | 可预测的状态管理 |
| 样式 | Tailwind CSS | 3.x | 原子化CSS、开发快 |
| 测试 | Jest | 28.x | 成熟、功能全 |

### 后端
| 层级 | 技术 | 版本 | 选择理由 |
|------|------|------|---------|
| 框架 | Express | 4.x | 轻量、灵活 |
| 数据库 | PostgreSQL | 13.x | ACID、生态成熟 |
| ORM | TypeORM | 0.3.x | TypeScript支持好 |
| 缓存 | Redis | 6.x | 高性能缓存 |
```

## ❓ 常见问题

**Q: 设计的粒度应该多细？**

A: 足以让 code-task 清晰地分解任务，不需要过度细致。通常到"模块"和"核心组件"层级就够了。

**Q: 应该包含多少性能设计？**

A: 根据规范的非功能需求而定。如果规范要求"API响应<200ms"，设计中应该有缓存、索引、查询优化等内容。

**Q: 设计完成后还需要验证吗？**

A: 建议让核心开发者review一遍，确保：
- 技术选型是否合理
- 架构是否清晰
- 是否有遗漏的地方

**Q: 可以修改设计吗？**

A: 可以，但应该记录：
- 什么改了
- 为什么改
- 对后续任务的影响
