## 审查清单

**代码质量：**
- 职责分离清晰？
- 正确的错误处理？
- 类型安全（如适用）？
- 遵循 DRY 原则？
- 边缘情况处理？

**架构：**
- 设计决策合理？
- 可扩展性考虑？
- 性能影响？
- 安全问题？

**测试：**
- 测试实际验证逻辑（而非模拟）？
- 边缘情况覆盖？
- 需要时是否有集成测试？
- 所有测试通过？

**需求：**
- 满足所有计划需求？
- 实现与规格一致？
- 没有范围蔓延？
- 是否有破坏性变更记录？

**生产就绪性：**
- 迁移策略（如有 schema 变更）？
- 考虑向后兼容性？
- 文档完整？
- 没有明显 bug？

## 关键规则

**应该：**
- 按实际严重程度分类（不是所有都是严重）
- 具体说明（文件:行号，不要模糊）
- 解释为什么问题重要
- 承认优势
- 给出明确结论

**不应该：**
- 不检查就说"看起来不错"
- 把小问题标为严重
- 对未审查的代码给出反馈
- 含糊其辞（"改进错误处理"）
- 避免给出明确结论
