# spec-archive

## 职责

将测试通过的需求规范合并到企业级主规范库中。

**核心流程**：
1. 分析需求Spec和主规范
2. 场景对标和差异分析
3. 规范合并和去重
4. 统一术语、数据模型、规则
5. 生成归档报告

**输出路径**：
- `spec-dev/spec/`（主规范目录）
- 可选：删除原需求目录（需要多层确认）

## 何时使用

**前置条件**：
- ✅ code-test 已完成，所有测试都通过
- ✅ code-execute 已完成，代码已生成
- ✅ 需求规范已验证（通过代码实现）

---

## 工作流程

### 步骤1: 分析需求Spec和主规范

#### 1.1 读取需求Spec结构

分析 `spec-dev/{requirement_desc_abstract}/spec/` 中的文档：
- 场景清单和数量
- 涉及的核心数据模型
- 业务规则和约束条件
- 新增的术语定义

#### 1.2 检查主规范

检查 `spec-dev/spec/` 是否存在：
- ✅ 如果存在 → 执行步骤2（合并）
- ✅ 如果不存在 → 执行步骤3（初始化）

### 步骤2: 场景对标和差异分析

对比新需求的场景与主规范中的已有场景：

| 分类 | 处理方式 |
|------|---------|
| **完全相同** | 标记为重复，去重合并 |
| **相关但不同** | 合并为通用场景的变体 |
| **全新场景** | 直接添加到主规范 |
| **特殊情况** | 作为异常处理场景 |

### 步骤3: 规范合并

#### 3.1 场景库合并

```
需求Spec的scenarios/
├── user-create-order.md       ┐
├── user-edit-order.md         ├─→ 合并到主规范的scenarios/
└── user-delete-order.md       ┘
```

处理方式：
- 相同的场景 → 去重，保留一份
- 类似的场景 → 合并为通用场景
- 新的场景 → 添加到库中

#### 3.2 数据模型合并

分析两个data-models.md的差异：
- **相同字段** → 去重
- **新增字段** → 添加到已有模型
- **新增模型** → 添加到库中
- **冲突** → 标记为需要解决的问题

#### 3.3 业务规则合并

分析两个business-rules.md的差异：
- **相同规则** → 去重
- **相关规则** → 合并或标记关联
- **新规则** → 添加到库中
- **冲突** → 标记为需要解决的问题

#### 3.4 术语表合并

统一glossary.md：
- **相同术语** → 保持一致定义
- **新术语** → 添加到库中
- **定义不同** → 标记为需要解决

### 步骤4: 一致性检查

生成一致性报告：

```
✅ 场景对标：18个新场景 → 12个已有 + 6个新增
✅ 数据模型：4个新模型 → 2个已有 + 2个新增
✅ 业务规则：8个新规则 → 6个已有 + 2个新增
✅ 术语表：15个新术语 → 10个已有 + 5个新增

⚠️ 需要人工解决的冲突：0
```

### 步骤5: 生成归档报告

生成报告到 `spec-dev/spec/archive-log.md`：

```markdown
# 规范归档报告

## 归档内容
- 需求ID：order-form (2026-03-26)
- 场景数：18
- 数据模型：4
- 业务规则：8
- 新术语：15

## 合并结果
- 新增场景：6
- 新增模型：2
- 新增规则：2
- 新增术语：5

## 一致性检查
✅ 所有场景都有对应的验收标准
✅ 所有模型都有清晰定义
✅ 所有规则都有应用场景
✅ 没有冲突需要解决

## 规范库当前状态
- 总场景数：42
- 总模型数：12
- 总规则数：18
- 总术语数：35

下次归档时间：[下一个需求的完成时间]
```

### 步骤6: 可选的清理

如果需要清理原需求目录（谨慎操作）：
- 获得用户多层确认
- 备份原目录
- 删除 `spec-dev/{requirement_desc_abstract}/`

## 关键约束

✅ **必须做**：
- 完整分析新需求的Spec
- 对标比较，找出重复和新增
- 处理所有冲突（标记或合并）
- 一致性检查通过
- 生成详细的归档报告

❌ **禁止做**：
- 随意删除已有规范
- 跳过重复检查
- 忽视冲突，直接覆盖
- 修改已有规范而不标记原因

---

## 最佳实践

1. **保留历史** - 所有归档都记录在archive-log.md中
2. **追踪溯源** - 每个场景都能追踪到哪个需求贡献的
3. **版本管理** - 定期备份主规范
4. **团队评审** - 重大合并需要团队确认
