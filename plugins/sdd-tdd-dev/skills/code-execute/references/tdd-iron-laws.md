# TDD 铁律

> 参考 superpowers:test-driven-development 最佳实践

## 铁律

```
无失败测试不写生产代码
```

**违反字面规则就是违反精神规则。**

## RED → GREEN → REFACTOR 严格循环

1. **RED**: 编写一个失败的测试
   - 只写一个测试，只测试一个行为
   - 必须看到测试失败（红色）
   - 如果测试没失败，说明测试有问题或功能已实现

2. **GREEN**: 编写最小量的生产代码使测试通过
   - 只写让测试通过的最少代码
   - 不要过度设计
   - 不要提前实现未测试的功能

3. **REFACTOR**: 改进代码质量
   - 测试必须保持通过状态
   - 消除重复、提高可读性
   - 每次只做小重构

## 违反即删除

**写了生产代码再写测试？删除重写。**

- 不要保留为"参考"
- 不要边看边写测试
- 不要"适配"它
- 删除就是删除

从测试重新开始。没有例外。

## 每个循环只测试一个行为

- 一个测试 = 一个行为
- 不要在同一个测试中测试多个事情
- 测试命名清晰描述预期行为
- 断言明确，不使用模糊的真假判断

## 常见诱惑与应对

| 诱惑 | 正确做法 |
|------|---------|
| "这个很简单，直接写实现" | 还是先写测试 |
| "TDD 太慢了，跳过" | 不跳过，测试是安全网 |
| "先写实现再补测试" | 删除实现，从测试开始 |
| "这个测试不好写" | 说明设计有问题，重新考虑设计 |
| "测试通过了，再多写点功能" | 停止，写新测试再实现新功能 |

## 覆盖率不是目标

- 目标：测试驱动设计，而非测试覆盖率数字
- 高覆盖率是 TDD 的副产品，不是目标
- 关注测试质量，而非数量
