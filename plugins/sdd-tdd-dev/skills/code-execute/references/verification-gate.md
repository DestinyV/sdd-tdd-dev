# 验证完成前门控

> 参考 superpowers:verification-before-completion 最佳实践

## 门控函数

```
IDENTIFY → RUN → READ → VERIFY → CLAIM
```

## 铁律

```
未运行验证就不能声称完成
```

## 门控流程

### 1. IDENTIFY（识别）
什么命令能证明这个 Task 完成？
- 明确验收标准
- 确定验证命令

### 2. RUN（运行）
执行完整命令（新鲜、完整）
- 不要依赖之前的运行结果
- 完整执行验证流程

### 3. READ（读取）
读取完整输出，检查退出码
- 不要只看最后几行
- 检查是否有警告或错误

### 4. VERIFY（验证）
输出是否确认了声明？
- 否：声明实际状态 + 证据
- 是：声明完成 + 附上证据

### 5. CLAIM（声明）
**只有在 VERIFY 通过后**才能声明完成

## 验证对照表

| 声称 | 必须提供 | 不足的证据 |
|------|---------|-----------|
| 测试通过 | 测试命令输出: 0 failures | 之前通过的记录、"应该通过了" |
| 编译成功 | 编译命令: exit 0 | Linter 通过、"看起来没问题" |
| 代码质量达标 | Lint + TypeScript 检查输出 | "遵循了规范" |
| Task 完成 | 验收标准逐项核对 + 证据 | "测试通过了" |
| SQL 执行成功 | 数据库执行结果 | "SQL 看起来没问题" |
| 接口联调通过 | 实际调用结果 | "代码写好了" |

## 红旗警告

- 使用"应该"、"大概"、"看起来" → **停止，去验证**
- 在验证前表达满意（"好的"、"完成了"、"没问题"）→ **停止**
- 准备提交但未验证 → **停止**
- 相信子代理的成功报告 → **必须独立验证**

## 不信任 Agent 报告

```
✅ Agent 报告成功 → 检查 git diff → 验证改动 → 报告实际状态
❌ 直接信任 Agent 报告
```

**底线**：运行命令 → 读取输出 → 然后声明结果。没有捷径。
