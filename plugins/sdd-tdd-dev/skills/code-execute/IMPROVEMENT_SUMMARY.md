# Git 分支安全确认 - 完整改进方案总结

**版本**：v1.0（2026-04-08）

---

## 改进概述

优化了 code-execute Skill 中关于 git 分支操作的确认机制。从最初的"只在创建 worktree 时确认"扩展到"全流程分支安全确认"。

### 改进范围

| 阶段 | 原来 | 改进后 |
|------|------|--------|
| **Worktree 创建** | 简单说明 | ✅ 执行前用户确认分支选择 |
| **代码提交** | 直接提交 | ✅ 提交前用户确认（新增） |
| **代码合并** | 无确认 | ✅ 合并前用户确认策略（新增） |
| **Worktree 删除** | 直接删除 | ✅ 删除前检查状态并确认（新增） |

---

## 核心改进：4 个关键场景

### 场景 1️⃣：Worktree 创建时的分支确认

**目的**：避免从错误的分支（如 master）创建 worktree

**流程**：
```
1. 检测当前分支
2. 使用 AskUserQuestion 向用户展示选项
3. 用户选择：[A] 当前分支 / [B] Master / [C] 其他
4. 创建 worktree
```

**用户界面**：
```
📋 Worktree基础分支

请选择Worktree创建时的基础分支？

[A] 当前分支（推荐）
    feature/fix_sdd_dev_plugin_0320

[B] Master分支
    master

[C] 其他分支
    输入自定义分支名称
```

---

### 场景 2️⃣：代码提交时的分支确认（⭐ 新增）

**目的**：避免提交到错误的分支

**流程**：
```
1. 检测当前分支
2. 显示待提交的改动
3. 使用 AskUserQuestion 向用户确认
4. 执行 git commit
```

**用户界面**：
```

当前分支：feature/fix_sdd_dev_plugin_0320
提交文件数：5
提交信息：

=====================================

确认代码提交信息？

[✓] 确认提交
    提交到当前分支：feature/fix_sdd_dev_plugin_0320

[?] 查看详情后确认
    先查看详细的改动后再确认

[✗] 取消提交
    暂不提交，继续修改
```

---

### 场景 3️⃣：代码合并时的确认（⭐ 新增）

**目的**：避免错误的合并策略或合并到错误的分支

**流程**：
```
1. 显示 worktree 的 commit 历史
2. 显示主分支的 commit 历史
3. 使用 AskUserQuestion 让用户选择合并策略
4. 执行合并操作
```

**用户界面**：
```
Worktree commit 历史：
- abc1234 feat(T1): UserAuth - 初始实现
- def5678 fix(T1): 修复Props接口

Main commit 历史：
- xyz9876 chore: 清除冗余文档

请确认代码合并的信息？

[◀] Cherry-pick（保留修复历史）
    推荐。将worktree的所有commit逐个合并到master，便于审计

[▬] Squash Merge（简洁）
    将所有worktree commit合并为一个，保持历史简洁

[✗] 取消合并
    暂不合并，继续修改或验证
```

---

### 场景 4️⃣：Worktree 删除时的确认（⭐ 新增）

**目的**：避免误删导致数据丢失

**流程**：
```
1. 检查 worktree 状态（未提交改动、临时文件）
2. 验证代码已合并到主分支
3. 使用 AskUserQuestion 向用户确认
4. 删除 worktree
```

**检查内容**：
```
========== Worktree 状态检查 ==========
✅ 无未提交的改动
✅ 所有commit已合并到master
✅ 无临时文件
======================================

确认删除Worktree？

[✓] 确认删除
    .claude/worktrees/T1-UserAuth
    所有改动已合并，可安全删除

[?] 查看详情后确认
    先检查worktree状态后再删除

[✗] 取消删除
    暂不删除，继续使用worktree
```

## 关键特性

### 1. 统一的用户交互工具
```
使用 AskUserQuestion 工具进行所有关键确认
不使用 shell 交互或提示文本
保证用户体验一致
```

### 2. 完整的分支追踪
```
执行报告中记录每个分支操作：
- 当前分支
- 用户选择
- 操作结果
- 时间戳
```

### 3. 安全的默认值
```
未选择时：自动使用当前分支（HEAD）
不假设任何分支名
避免 "Not a valid object name" 错误
```

### 4. 清晰的错误处理
```
分支不存在：显示可用分支列表
提交失败：记录错误并提示用户
合并冲突：指导用户手动解决
```

## 常见问题

**Q1: 为什么需要在每次提交时确认？**
A: 避免在 worktree 中不小心切换到其他分支后继续提交。AskUserQuestion 确保用户始终清楚当前分支。

**Q2: 如果多个Task都选择同一分支，是否重复确认？**
A: 第一个Task询问用户，后续Task继续使用相同分支，无需重复询问。可在执行报告中清晰记录。

**Q3: 如何处理合并冲突？**
A: 提示用户手动解决冲突后重新执行合并。在执行报告中记录冲突和解决方案。

**Q4: 删除Worktree时发现未合并的改动怎么办？**
A: 提示用户，不执行删除。让用户选择：重新审查代码、手动合并、或保留worktree。
